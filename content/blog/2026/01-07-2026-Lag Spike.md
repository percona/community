

<a name="_aeu4fn9gseql"></a>How I Debugged a Replication Lag Spike in PostgreSQL Using Percona Toolkit

## <a name="_tluvz03q55yv"></a>Introduction

We have all been there. It is 2:00 PM on a Tuesday, and the application starts timing out. Users are complaining. The monitoring dashboard is lighting up like a Christmas tree. And somewhere, deep in the stack, a PostgreSQL replica is struggling to keep up.

That was my situation last quarter. A production database with a long-standing replication lag spike, measured in hours, not minutes. The primary was humming along, but the replica was stuck in the past. Reporting queries were failing. Read replicas were useless. And nobody knew why.

This is the story of how I used pt-query-digest from Percona Toolkit to find the root cause and fix it, without downtime. I share it because replication lag is one of those problems that can feel like a mystery until you have the right approach. The tools exist. The patterns are proven. And with a little patience, you can solve it.

## <a name="_gmfdn8td3txz"></a>The Symptoms

The first signs were subtle. A few application timeouts. Some users complaining about slow dashboards. Then the alerts started: replication lag on the primary replica had exceeded 10 minutes. Then 30. Then an hour.

I logged into the primary and ran the usual checks:

sql

SELECT pid, application\_name, state, sync\_state, write\_lag, flush\_lag, replay\_lag

FROM pg\_stat\_replication;

The lag was real. The output showed replay\_lag climbing steadily. But the hardware looked fine. CPU and memory usage were normal. Network latency was low. The database itself was not under heavy write load. The primary was not struggling. The replica was not failing. It was just lagging behind, and the gap kept growing.

I checked the replication slot. I checked the WAL archiving. I restarted the replica. Nothing changed. The lag was persistent and consistent. It was not a transient spike. It was a steady, grinding delay that made read replicas useless.

The problem was not obvious. It was hiding somewhere else, buried in the work the replica had to replay.

## <a name="_yeow1dwp6yal"></a>Using pt-query-digest to Find the Problem

I had used Percona Toolkit before, mostly for schema changes and backups. But pt-query-digest is something else entirely. It analyzes slow query logs and turns them into something you can actually read. It does not just show you slow queries. It shows you the patterns, the outliers, and the queries that are doing the most damage.

First, I enabled slow query logging on the primary. I wanted to capture the queries that were running during the lag window, not just the daily traffic. I set the threshold to 500 milliseconds, lower than my usual standard, because I wanted to see everything that could be contributing.

sql

ALTER SYSTEM SET log\_min\_duration\_statement = 500;

SELECT pg\_reload\_conf();

Then I waited. A few minutes later, I grabbed the slow query log and ran the tool:

bash

pt-query-digest /var/log/postgresql/postgresql-slow.log > slow-query-analysis.txt

The output was clean, structured, and terrifying. Pt-query-digest

ranked the queries by total time, call count, and average latency. One query stood out immediately. It was running hundreds of times per minute. Its execution time was high, and its call count was enormous. The tool flagged it as the top query by total time and latency. The culprit was a simple SELECT on a high-volume orders table. It was not a complex join or a recursive CTE. It was a basic filter on a column that had no index.

The query was trying to find recent orders for a reporting dashboard. The dashboard pulled data every few seconds, and each request scanned the entire table. Over time, the scans grew longer as the table grew larger. And the replica had to replay every single one of them.

## <a name="_t9w9xessffbo"></a>The Root Cause

The query was filtering on created\_at, a column with no index. The table had grown to over 50 million rows, and the query was performing a full sequential scan every time it ran. That meant the primary had to scan hundreds of thousands of rows per execution, and the replica had to replay all of that work.

The result was a cascade. The primary processed the query, the replica tried to replay the same work, and the I/O overhead saturated the replica. The lag grew and grew until it broke. The replica was not failing. It was simply overwhelmed by the volume of work it had to replay. It could not keep up with the primary, and the gap widened with each passing minute.

The fix was obvious. But the challenge was applying it without taking the database offline. Adding an index to a 50-million-row table in production is not something you do during business hours without a plan.

## <a name="_c1n3mxm4ki6y"></a>The Fix

I used pt-online-schema-change, another Percona Toolkit tool, to add the index without blocking writes:

bash

pt-online-schema-change --alter "ADD INDEX idx\_orders\_created\_at (created\_at)" --execute D=database,t=orders

The tool works by creating a new table, applying the schema change, copying data in chunks, and then swapping the tables. It is safe, tested, and widely used. It does not lock the table, so writes continue uninterrupted. It also has safeguards like chunk size limits and foreign key handling to prevent accidents.

The operation took about 15 minutes. There was no downtime. Application writes continued without interruption. The application did not even notice the change.

After the index was created, the query execution time dropped by 95 percent. Replication lag returned to normal within minutes. The alerts stopped. The application recovered. The dashboard that had been loading slowly for days was suddenly snappy again.

## <a name="_k7ub72a7yalw"></a>Lessons Learned

This incident taught me a few things that I carry with me today.

The first is that monitoring replication lag is not enough. You need to know what is causing it. Lag is a symptom, not a root cause. Without pt-query-digest, I would have been chasing ghosts for days. I would have blamed the network, the hardware, or the WAL configuration. None of those were the problem.

The second is that missing indexes are often the root of replication lag. Queries that are tolerable on the primary can be catastrophic on a replica. A scan that takes 100 milliseconds on the primary can take seconds to replay on the replica. And when that query runs hundreds of times per minute, the replica simply cannot catch up.

The third is that Percona Toolkit is not just a set of scripts. It is a safety net. pt-query-digest showed me where to look. pt-online-schema-change let me fix it without fear. I could make the change confidently, knowing that the tool would handle the heavy lifting and keep the database available.
